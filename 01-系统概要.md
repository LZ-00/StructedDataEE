## 1 引言

### 1.1 编写目的

本操作手册用于说明**激光焊接全流程数据库管理系统**（以下简称「本系统」）的安装、配置、功能模块及日常操作方法，帮助操作人员、工艺工程师及系统维护人员正确、安全地使用本系统。

编写本手册的目的在于：

1. 明确本系统的建设背景、定位与建设目标，便于用户理解系统在全流程业务管理中的作用；
2. 概括系统功能结构与各子模块职责，为后续分模块操作说明提供总览；
3. 规定系统运行所需的硬件、软件及网络环境，保障系统稳定部署与运行；
4. 统一相关专业术语与缩略语，便于阅读与沟通。

### 1.2 适用范围

本手册适用于本系统的以下人员：

- **系统操作人员**：负责登录系统、执行文献解析与结构化抽取、查看评估结果、管理模型配置等日常操作；
- **工艺管理人员**：负责校对蒸馏样本、组织模型评测与迭代；
- **系统维护人员**：负责前后端服务部署、运行环境配置、模型权重维护。

本手册描述的系统版本为 **V1.0**，对应软件实现为前后端分离架构：前端基于 Vue 3 + Element Plus，后端基于 FastAPI，业务模块包括智能抽取、思维链评估、蒸馏与 LoRA 微调、模型注册管理等。

### 1.3 定义和缩略语

| 术语/缩略语 | 说明 |
|------------|------|
| 激光焊接全流程 | 涵盖工艺文献/试验报告等非结构化文本的采集、结构化入库、质量评估、蒸馏构建及领域模型迭代等环节 |
| 结构化抽取 | 从非结构化文本中识别实体（如材料、工艺参数、缺陷类型等）及实体间关系，输出规范化 JSON 结构 |

| API / 本地模型 | 分别指通过 OpenAI 兼容接口调用的远程大模型，以及通过 ModelScope 部署在本地的大语言模型 |

| RMSE | 均方根误差，用于衡量预测与金标准在数值维度上的偏差 |

## 2 系统概述

### 2.1 背景

激光焊接工艺知识广泛存在于期刊论文、工艺卡片、试验记录、检测报告等非结构化文档中。传统管理方式依赖人工摘录与表格录入，存在效率低、口径不统一、难以与智能分析工具衔接等问题。随着大语言模型在信息抽取与推理任务上的发展，将**文献解析—结构化入库—质量评估—蒸馏构建—模型微调**纳入统一平台，已成为构建激光焊接全流程数据库的重要路径。

本系统面向上述需求，以 Web 方式提供从非结构化文献到结构化工艺信息、再到模型评测与迭代的全链路支撑，为铝合金、不锈钢、钛合金等多类材料的激光焊接场景提供可复用的知识与模型管理基础。

### 2.2 系统定义

**激光焊接全流程数据库管理系统**是一套 B/S 架构的业务软件。系统通过浏览器访问，由前端交互界面与后端 REST API 协同工作，围绕激光焊接领域的**非结构化文本智能化处理**与**结构化工艺知识管理**展开，核心能力包括：

- 对 PDF/TXT 工艺报告及纯文本输入进行**结构化关系抽取**；
- 基于金标准进行**思维链（CoT）驱动的质量评估**；
- 利用教师模型进行 **CoT 蒸馏**并支持 **LoRA 微调与模型发布**；
- 提供**模型注册表**统一管理 API 与本地大模型；
- 提供**用户登录与 JWT 鉴权**，保障接口访问安全。

系统界面标题为「激光焊接结构化数据抽取与质量评估系统」，作为全流程数据库管理中面向智能抽取与质量评估的核心子系统实现。

### 2.3 建设目标

本系统建设目标如下：

1. **信息结构化**：将激光焊接文献中的基材、工艺参数（功率、速度、保护气体等）、缺陷与质量结果等要素抽取为统一的实体—关系结构，支撑工艺知识库建设；
2. **质量可度量**：自动评估抽取效果，输出精确匹配率、RMSE 等指标，并以四步思维链（解析—验证—诊断—统合）呈现推理过程；
3. **知识可迭代**：通过教师模型蒸馏生成高质量 CoT 样本，经校对后用于 LoRA 微调，持续提升领域抽取模型能力；
4. **资源可配置**：集中管理 Qwen、Llama 等本地模型及 GPT-4o 等 API 模型，支持推理检测、权重下载与启用/禁用；
5. **操作可落地**：采用前后端分离与模块化 API 设计，便于在实验室内网环境部署与扩展。

### 2.4 系统特点

1. **全流程闭环**：覆盖「文献/文本输入 → 结构化抽取 → 质量评估 → 蒸馏构建 → 微调发布」完整链路；
2. **双通道输入**：智能抽取工作台支持文件上传（PDF/TXT）与文本直接输入，并支持自定义任务描述（Instruction）；
3. **可视化评估**：思维链评估模块提供状态机流转、CoT 日志面板及结果判定卡片，便于分析模型行为；
4. **灵活模型栈**：同时支持本地 ModelScope 部署与 OpenAI 兼容 API，教师/学生模型可按场景选配；
5. **轻量微调**：集成 PyTorch、Transformers、PEFT 等组件，支持 LoRA 微调。

### 2.5 系统技术架构概要

- **前端**：Vue 3 + TypeScript + Vite + Element Plus + ECharts，单页应用路由守卫实现登录态校验；
- **后端**：FastAPI 模块化路由——`/api/auth`、`/api/extraction`、`/api/evaluation`、`/api/distillation`、`/api/finetune`、`/api/models`；
- **业务逻辑**：集中于 `backend/app/services/`，配置集中于 `backend/app/config/`（模型目录等）；
- **部署方式**：项目根目录 `npm run dev:backend` 与 `npm run dev:frontend` 分别启动前后端，详见 `DEVELOPMENT.md`。

## 3 系统功能结构

### 3.1 功能总体结构

本系统采用「**表示层 — 业务服务层 — 模型层**」三层结构：

```
┌─────────────────────────────────────────────────────────────┐
│                    浏览器客户端（Vue 3）                      │
│  登录 │ 智能抽取工作台 │ 思维链评估 │ 蒸馏与微调 │ 模型配置   │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP /api（JWT）
┌──────────────────────────▼──────────────────────────────────┐
│                  业务服务层（FastAPI）                        │
│  认证 │ 抽取 │ 评估 │ 蒸馏 │ 微调 │ 模型注册表                  │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                      模型层                                   │
│                    模型权重与推理服务                          │
└─────────────────────────────────────────────────────────────┘
```

用户经登录页完成身份认证后，进入主界面顶部导航，按业务需要切换各功能模块；后端通过 `/api` 前缀提供统一接口，前端开发环境下由 Vite 将 `/api` 代理至后端 8000 端口。

### 3.2 主要功能说明

| 序号 | 功能模块 | 功能说明 |
|------|----------|----------|
| 1 | 用户登录与权限 | 提供用户名/密码登录，签发 JWT；未登录用户无法访问业务页面；支持退出登录 |
| 2 | 智能抽取工作台 | 选择抽取基座模型；上传 PDF/TXT 或输入文本；配置任务描述；一键执行抽取；左右分栏展示原文与结构化 JSON（实体、关系）；支持导出 |
| 3 | 思维链评估 | 选择评估模型；启动评估；四步状态机展示；CoT 日志实时查看；输出精确匹配率、RMSE 等综合判定 |
| 4 | 蒸馏与微调平台 | **步骤一**：选择教师模型与系统指令，一键生成 CoT 蒸馏结果并支持预览校对；**步骤二**：基于校对结果配置 LoRA 微调并运行；**步骤三**：微调完成后发布模型供抽取/评估选用 |
| 5 | 模型配置管理 | 维护模型注册表（API/本地）；添加、编辑、删除、启用/禁用；本地模型支持 ModelScope 下载与推理检测 |

**结构化抽取输出**主要包括：`document_id`、实体列表（文本、类型、位置）、关系列表（类型、源/目标实体及类型、证据文本），用于描述「工艺参数—缺陷/质量结果」等领域关系。

## 4 运行环境

### 4.1 硬件环境

| 项目 | 最低配置 | 推荐配置 |
|------|----------|----------|
| 处理器 | 64 位多核 CPU | 8 核及以上 |
| 内存 | 16 GB | 32 GB 及以上（本地 7B 级模型推理/微调） |
| 硬盘 | 50 GB 可用空间 | 100 GB 及以上（含 ModelScope 模型权重） |
| 显卡 | 无（仅 Web 与 API 服务，不含本地大模型推理/微调） | NVIDIA GPU，显存 ≥ 16 GB（LoRA 微调及本地大模型推理） |
| 显示器 | 分辨率 1920×1080 | 同左 |

### 4.2 软件环境

| 类别 | 要求 |
|------|------|
| 操作系统 | Linux（推荐 Ubuntu 20.04+）或 Windows 10/11、macOS（开发调试） |
| 后端运行环境 | Python 3.9 及以上；依赖见 `backend/requirements.txt`（FastAPI、Uvicorn、PyJWT、PyTorch、Transformers、PEFT、ModelScope 等） |
| 前端运行环境 | Node.js 18 及以上；npm；依赖见 `frontend/package.json`（Vue 3、Vue Router、Element Plus、ECharts、Axios 等） |
| 浏览器 | Chrome、Edge、Firefox 等现代浏览器（建议最新稳定版） |
| 可选配置 | `backend/.env` 中配置 `SECRET_KEY`、`DEFAULT_USERNAME`、`DEFAULT_PASSWORD`；本地模型默认目录 `/data/lz/modelscope`（可在模型配置中修改） |

### 4.3 网络环境

- 本地部署：前端默认 `http://localhost:3000`，后端 `http://127.0.0.1:8000`，前端通过代理访问 `/api`；
- 调用 API 类教师模型时需能访问相应 OpenAI 兼容服务端点；
- 下载 ModelScope 本地权重时需能访问模型托管网络；
- 建议部署于实验室内网；生产环境应配置 HTTPS、修改初始账号密码并限制 CORS 来源。
